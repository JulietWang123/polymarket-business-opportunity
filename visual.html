01 / CORE JUDGMENT

预测市场的价值正在从交易转向概率基础设施。

Polymarket 的主流化不只来自交易热度，更来自事件概率数据、媒体引用、合规化路径和 AI Agent 可调用场景。

核心资产 事件概率数据网络 交易是起点，长期价值在于高频概率信号、历史数据、媒体引用和机构/Agent 可调用能力。
主流化路径 合规与机构分发 Polymarket US、ICE、MLB 等合作显示预测市场正在靠近合规金融、体育娱乐和机构数据分发。
必要约束 价格不是绝对真相 市场价格有信息价值，但会受流动性、参与者结构、操纵风险和结算歧义影响。
风险边界 信息层优于交易层 创业团队更适合做只读数据、解释、风控和审计；自动下单与自建撮合会显著抬高风险。
产品机会 从概率到工作流 概率 API、变化解释、结算风险评分和告警系统能嵌入媒体、研究、交易和 Agent 工作流。
AgentPort 定位 概率与风险判断层 AgentPort 应先验证预测数据调用和付费，再逐步扩展为 Web3 服务发现与聚合平台。

02 / MARKET STRUCTURE

Polymarket 的护城河不是链，而是市场质量。

预测市场的价格是一种有噪声的概率信号。它的价值来自流动性、市场运营、结算可信度和被媒体/机构/Agent 反复引用的习惯。

为什么预测市场主流化

  • 事件合约进入新闻叙事：选举、利率、体育、战争、公司事件都适合概率化表达。
  • 传统预测信任下降：用户更愿意看“用钱表达”的连续价格。
  • 加密基础设施成熟：稳定币、低成本链、链下撮合降低交易摩擦。
  • AI 需要结构化概率：Agent 做规划、研究和风控时，概率比新闻标题更适合进入决策链。

Polymarket 的真实优势

流动性网络效应：热门市场的深度、价差和成交速度形成惯性
品牌心智：媒体开始把 Polymarket 当作事件概率参考
市场运营：标题、结算条件、分类和激励决定市场质量
数据资产：历史价格、成交和结果可用于校准与模型训练
合规转型：Polymarket US 提升机构可接受度

03 / BUSINESS MODEL

交易费只是入口，事件概率数据才是长期资产。

如果 Polymarket 只是撮合用户交易，天花板取决于交易量和费率；如果它成为金融终端、媒体、体育联盟和 Agent 共同引用的概率源，商业空间会更大。

收入方向说明成熟度
交易费部分市场 taker fee，配合 maker/taker rebate 改善流动性。已开始
数据分发事件概率、历史数据和实时价格可进入金融终端、媒体和机构产品。高潜力
API 与开发者服务市场数据、订单簿、结算结果、事件分类等可形成开发者产品。中高
机构风控工具异常波动、鲸鱼交易、结算争议、内幕交易线索监控。中高
体育/娱乐合作官方数据、品牌曝光、完整性框架和粉丝互动。高
白标与基础设施为媒体、交易平台和 Agent 平台提供概率组件。中

04 / RISK LAYER

监管风险不是开关，而是阶梯。

只读数据和研究分析相对安全；交易信号、自动下单、资金托管和自建撮合会显著抬高合规、安全和声誉成本。

产品层级风险等级说明
只读公开数据聚合低接近信息服务，适合 MVP 起步。
概率解释与研究分析低到中不提供收益承诺和交易建议时风险可控。
交易信号与策略建议中到高可能触及投资建议、博彩推荐和适当性问题。
自动下单/代客交易高涉及牌照、KYC、授权、安全和地域限制。
自建撮合交易市场极高需要处理交易所、衍生品、博彩、支付和市场监管问题。

05 / OPPORTUNITIES

不要复制 Polymarket，站到它的上游和旁路。

更好的机会在信息层、解释层、审计层和 Agent 工具层，而不是直接做一个交易市场。

01Prediction API聚合 Polymarket、Kalshi、新闻、宏观数据和赔率，输出结构化概率、来源和置信度。
02市场解释层解释概率为什么变化，把价格波动转成中文因果链。
03结算规则审计识别市场标题、数据源、截止时间和 oracle 争议风险。
04风控 SaaS监控鲸鱼账户、异常波动、疑似内幕交易和跨市场套利。
05中文事件概率覆盖华语宏观、公司、文化和体育事件，只做信息服务，不撮合交易。

06 / AGENTPORT

AgentPort 应先做概率与风险判断层。

AgentPort 的方向成立，但早期不要做大而全的 Web3 API marketplace。先用预测市场数据做可验证、可付费、可嵌入 Agent 工作流的垂直产品。

推荐定位

AgentPort 是 Agent 可调用的概率与风险判断层。预测市场数据是入口，x402 是支付能力展示，长期目标是 Web3 服务发现与调用网络。

07 / GO TO MARKET

先服务真实调用，而不是先搭平台。

推广重点不是“我们支持 x402”，而是“Agent 可以用一个接口得到事件概率、变化原因和风险评分”。

用户痛点切入方式
Web3 研究/交易 Agent 开发者需要实时概率、事件搜索、风险评分。MCP Server、SDK、示例 Agent。
中文加密研究团队/KOL/媒体需要把英文预测市场转成中文解释和图表。事件页、嵌入组件、每日概率变化。
小型基金/做市/风控团队需要异常监控、回测和风险预警。Dashboard、API、企业试用。
Agent 平台和钱包需要 x402 机器支付 demo 和高频工具。联合 demo、生态合作、收入分成。

08 / ROADMAP

四阶段路线图。

每个阶段只推进一个核心验证：先验证调用，再验证质量，再验证付费，最后才验证平台化。

PHASE 0 · 4-6 周垂直 MVP

Prediction API、MCP Server、x402 demo、10-20 个高质量事件页。不做交易执行，不开放 marketplace。

PHASE 1 · 2-3 月质量比较

多源概率比较、Brier score、结算风险评分、概率变化解释和异常提醒。

PHASE 2 · 3-6 月企业场景

风控 Dashboard、机构 API、团队权限、审计日志和定制事件池。

PHASE 3 · 6-12 月平台开放

第三方服务商 SDK、服务质量认证、x402 分账结算和更多 Web3 数据品类。

最终判断

不要先做 Web3 版 RapidAPI，也不要做中文版 Polymarket。先做 Agent 可调用的概率与风险判断层，让 Agent 用一个接口获得未来事件概率、变化原因和可信度评估。
